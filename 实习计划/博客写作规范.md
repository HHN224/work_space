# 博客写作规范

> 本规范约束 `实习计划/` 仓库内所有面向半公开学习日志的技术内容。
> 目标读者：未来的自己 + 同行 + 面试官。形态：学习日志 + 技术，保留真实心路但克制，不写裸情绪。

---

## 一、五条写作原则

1. **一文一事** —— 每道 LeetCode 题、每个八股知识点独立成篇。周报（`weekly/`）只做索引，不承载正文。
2. **先结构后文字** —— 动笔前先套模板的固定模块，再往里填内容。模块齐全优先于文采。
3. **情绪必须转化** —— 裸情绪句（"我累死了""我太菜了""气笑了""不适合学计算机"）禁止入正文。所有挫败感/吐槽统一进「踩坑反思」区块，按下文 4 步结构改写。
4. **复杂度必填** —— 题解无时间/空间复杂度分析视为未完成，不得发布。
5. **可迁移 > 单题** —— 题解每篇必须收口"这类题的通用规律"；只讲清这一道题、给不出可复用套路就是不合格。
6. **八股优先短而真** —— 八股不追求模块齐全。一个知识点最多学两天，撑不起一堆模块，强行凑齐只会变成 AI 味的百科词条。八股只要求两件套（一句话定义 / 我的理解一段话），代码和拓展区都可选——概念性八股不贴代码很正常。写不出就如实写"还没完全懂"，**绝不用 AI 补全**。

## 三、元数据规范（frontmatter）

每篇正文文件以 YAML frontmatter 开头。

**题解**：
```yaml
---
题号: 15
题目: 三数之和
难度: Medium
分类: [数组, 双指针]
链接: https://leetcode.cn/problems/3sum/
日期: 2026-06-15
状态: 看题解后重写   # 独立解出 / 看题解 / 看题解后重写
---
```

**八股**：
```yaml
---
主题: GIL 全局解释器锁
分类: python
日期: 2026-06-15
关联面试题: [Python 多线程为什么不能并行, CPU 密集 vs IO 密集怎么选]
---
```

`状态` 字段诚实填写，是给自己复盘用的，不丢人。

## 四、语言与排版规范

- **全中文为主**，英文题面、专有名词、代码标识符原样保留（如 `__call__`、`asyncio`、`GIL`）。
- **题面复述，不照抄原文**：用自己话 2-3 句说清输入/输出/约束，证明你读懂了。照抄英文原文无价值。
- **代码块必标语言**：` ```python `，不要裸 ` ``` `。
- **注释密度**：关键行（核心状态转移、指针移动条件、退出判断）必须有注释，废话注释删掉。
- **第一人称使用边界**：可以用"我"，但仅限描述思路演进（"我先尝试了暴力"），不用于情绪（"我快崩溃了"）。
- **对比用表格**：浅拷贝 vs 深拷贝、TCP vs UDP、列表 vs 生成器等，优先用表格而不是大段文字。
- **复杂度写法**：`时间 O(n²) / 空间 O(1)`，并说明 n 是什么。

## 五、情绪转化规则（核心）

任何一句你想写的"我太菜了/我累了/气笑了/不适合学计算机"，都必须按下表转化为结构化反思，放进「踩坑反思」区块。

**4 步结构**：

1. **错误假设**：我当时的错误认知/假设是什么
2. **为什么错**：这个假设哪里不成立
3. **纠正认知**：正确的理解是什么
4. **下次识别信号**：再遇到什么特征，就该直接想到正解

**对照表**：

| 原文（裸情绪，禁止） | 改写（踩坑反思，要求） |
|---|---|
| "做这题真力竭了…一看题解气笑了居然要二维数组…突然觉得自己不适合学计算机" | **错误假设**：以为 LCS 能用一维 DP。**为什么错**：两串各有一个游标，状态依赖两维位置。**纠正**：双序列 DP 必开二维。**下次信号**：题面给"两个序列/字符串"且问"公共/匹配/编辑"→ 直接上二维 DP。 |
| "今天有点累，就随便写写了" | （整句删除。或转为：本解法为暴力多源 BFS 变体，更优解见拓展——标准 BFS 队列） |
| "我一开始写了个死循环调试半天" | **错误假设**：循环条件只写 `while fast`。**纠正**：fast 走两步需保证 `fast and fast.next`。**下次信号**：快慢指针一律先判 `fast.next`。 |
| "作为人类实在是太难想递归了" | **错误假设**：想直接脑算整条递归链。**纠正**：先定"栈顶的特殊返回值"，再假设倒数第二层拿到返回值后做什么。**下次信号**：树/链表递归题先写 base case 的返回值。 |

**判断标准**：一句话如果删掉对读者理解题目/知识点没有任何损失，就该删掉或转化。

## 六、质量自检清单（发布前逐条勾选）

自检分两套，按内容类型选一套。

### 题解自检

- [ ] frontmatter 元信息齐全
- [ ] 题面是复述，不是照抄原文
- [ ] 代码块标了语言、关键行有注释
- [ ] 复杂度分析已写（时间 + 空间 + n 的含义）
- [ ] 「踩坑反思」区块存在，且按 4 步结构写
- [ ] 正文无裸情绪句（用 Ctrl+F 搜：累、菜、气、崩溃、废、放弃）
- [ ] 「可迁移规律」已收口，给出了题型归类或套路
- [ ] 关联知识点用 `[[name]]` 链接了相关篇目
- [ ] 文件名符合命名规范
- [ ] 对应周报索引已更新（链接到本篇）

### 八股自检（精简版，门槛低）

- [ ] frontmatter 三字段（主题 / 分类 / 日期）
- [ ] 一句话定义是**自己的话**，不是抄来的
- [ ] 「我的理解」是一段自己的推理，不是定义堆砌
- [ ] 正文无裸情绪句
- [ ] **没有用 AI 补全正文**（写不出就如实写"还没完全懂 X"）
- [ ] 代码和拓展区都没写也没关系——概念性八股不贴代码很正常，不强制

## 七、周报索引规范

周报（`weekly/weekXX.md`）**不再是内容载体**，只做四件事：

1. **本周概览**：2-3 句
2. **完成题目**：表格（题号 | 题目 | 难度 | 状态 | 链接到单篇）
3. **八股笔记**：列表，链接到单篇
4. **本周踩坑 Top 3**：从各篇「踩坑反思」提炼最高频的 3 条
5. **下周计划**：对齐 `Study_Plan.md` 当周目标

每周写完后，先在 `posts/` 下把每道题/每个知识点成篇，再来填周报索引。顺序反了就是本末倒置。

## 八、与现有文件的关系

- `week1.md` / `week2.md` / `week3.md`：原始草稿，保留作对照，文件头标注迁移指引。正文以 `posts/` 和 `weekly/` 为准。
- `Study_Plan.md`：学习计划源，周报的「下周计划」对齐它。
- `LRU_cache.py`：长代码附件，题解里用相对路径链接引用，不内联。
